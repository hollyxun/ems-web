# 先查项目惯例再动手

## 背景

在项目中添加新功能（如新增页面、新增组件、新增 API 封装）时，如果不先了解项目已有的惯例，容易创建出与项目风格不一致的实现，事后返工。

## 内容

1. **加新功能前先查看同类功能的历史实现**
   - 去 Git 历史中搜索类似的 PR 或 commit
   - 查看已有组件的实现方式（如 `src/components/common/`）
   - 了解项目惯例

2. **优先复用已有模式**
   - 如果项目已有统一的 API 封装方式、组件组织方式、页面结构
   - 遵循已有模式而非另起炉灶

3. **不确定时问**
   - 如果找不到先例，先和用户确认方案再动手

## EMSS Web 具体实践

### 新增页面前
查看 `src/views/` 目录：
- 页面组件是否使用 `<script setup lang="ts">`？
- 如何处理 loading 状态？
- 如何使用 Element Plus 组件？
- API 调用如何组织？

### 新增组件前
查看 `src/components/common/` 目录：
- Props 如何定义？使用接口还是类型别名？
- 事件如何定义？使用 `defineEmits` 的类型化语法？
- 是否使用 `v-bind="$attrs"` 透传属性？
- 样式使用 UnoCSS 还是 scoped CSS？

### 新增 API 封装前
查看 `src/api/` 目录：
- 请求参数类型如何定义？
- 响应数据类型如何定义？
- JSDoc 注释格式？
- 错误处理如何统一？

## 示例

**正面**：添加新页面前，先查看 `src/views/system/user/index.vue`，发现统一使用组合式函数 `useTable` 处理表格逻辑，于是遵循相同模式。

**反面**：没查历史就创建了 Options API 风格的组件，与项目 Composition API 风格不一致，最终需要重构。
